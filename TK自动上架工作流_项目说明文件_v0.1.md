# TK 平台自动上架工作流
**供 Codex + OpenClaw 执行的项目说明文件（MVP 版）**

版本：v0.1  
项目负责人：TRED  
目标：先跑通 0-1，再逐步放量

## 1. 项目目标与业务边界

### 1.1 项目目标
1. 基于单个产品资料包，自动生成商品卡所需素材，包括主图、副图、标题、详情页文案、A+图、属性、SKU、价格等。
2. 通过飞书完成关键节点的人审，避免 AI 错误内容直接上架。
3. 审核通过后，调用上架流程写入对应店铺，并把链接回传给运营复检。
4. 优先做 MVP，只跑通 1 个类目、1 个站点、1 套模板、1 个上架入口。

### 1.2 不建议一开始就全自动的环节
- 主图最终确认：必须保留人工审核。
- 价格与利润：必须由公式或模板驱动，不建议让大模型自由发挥。
- 类目属性映射：必须基于 TikTok 官方模板/枚举值，不建议让 AI 自填开放文本。
- 正式发布：建议先“草稿上架”或“待发布状态”，最后由运营点确认。

## 2. 你当前工作流的主要问题与修正建议

### 2.1 结论
你的大方向是对的，但需要额外补上五个模块：
- 结构化输入
- 多级审核
- 发布前校验
- 失败回滚
- 人工接管点

### 2.2 对原工作流的修正
1. **图片先生成再审**：正确，但要拆成“主图审核”和“副图/A+审核”两级。
2. **标题在图片通过后再生成**：合理，标题依赖主图风格、卖点表达、目标站点语言。
3. **详情页内容与 A+ 图一起生成**：可以，但建议先生成详情文案结构，再生成 A+ 图。
4. **审核通过后自动上架**：建议改为“生成结构化上架包 -> 预校验 -> 调用上架 -> 返回草稿/链接 -> 运营复核”。
5. **填写 attribute / sku / price**：必须前置建模，它们不是“补填板块”，而是独立模块。
6. **通知运营检查链接**：正确，但需要同时回传店铺、产品、模板版本、价格、SKU、生成时间、任务日志。

## 3. 推荐的自动上架总流程（标准版）
1. 产品资料入库：收集产品图、规格、卖点、成本、目标站点、目标店铺、类目、禁限词要求、竞品链接/图。
2. 预处理：标准化命名，检查资料是否齐全，生成唯一任务 ID。
3. 生成商品图片：先出 1 张主图，再出 8 张副图/A+素材，不合格时允许局部重生成。
4. 主图人工审核：飞书通知运营，只审核主图和首屏卖点。
5. 生成标题：调用标题模板，根据站点语言、字符限制、核心关键词输出 3-5 个候选。
6. 生成详情页文案：输出卖点版块、参数版块、适用人群、使用方式、注意事项、FAQ。
7. 生成 A+ 图：基于详情页结构生成图文素材，确保与文案一一对应。
8. 二次人工审核：飞书审核标题、详情文案、A+ 图。
9. 结构化上架包组装：生成 attributes、SKU、price、库存、物流、类目字段。
10. 预校验：校验字符长度、必填字段、价格区间、图片数量、文件尺寸、属性枚举值。
11. 调用上架动作：优先走草稿；若平台接口不支持，则走模拟操作但保留截图和日志。
12. 回传结果：飞书通知运营检查最终链接，并保存全量日志。

## 4. 模块化拆分（适合 Codex + OpenClaw）

### 模块 A：任务入口模块
- 读取产品资料包
- 创建任务 ID
- 分配站点 / 店铺 / 模板版本

### 模块 B：素材生成模块
- 生成主图、副图、A+图
- 记录 prompt、版本号、输出文件路径

### 模块 C：人工审核模块
- 飞书发送卡片
- 接收“通过 / 驳回 / 重做原因”
- 回写状态

### 模块 D：文案生成模块
- 生成标题、卖点、详情、FAQ、短描述
- 输出结构化 JSON

### 模块 E：属性映射模块
- 把产品资料映射成 TikTok 模板字段
- 不允许自由文本乱填

### 模块 F：价格计算模块
- 根据成本、运费、抽点税率、目标 ROI 生成建议售价和利润表

### 模块 G：上架执行模块
- 把 package 写入平台
- 失败时截图、记录 DOM/接口返回

### 模块 H：结果回传模块
- 飞书推送链接、日志、异常原因、待人工处理项

## 6. 各阶段输入 / 输出 / 审核点
- 产品资料入库：输入产品图、规格、成本、卖点、站点、店铺；输出标准化任务包；审核资料是否齐全。
- 主图生成：输入任务包 + 主图规则；输出主图 1-3 版；审核主图。
- 副图/A+生成：输入主图风格 + 详情框架；输出副图 8 张 + A+素材；审核图文一致性。
- 标题生成：输入关键词、卖点、站点语言；输出 3-5 个候选；审核字符数/违禁词。
- 详情文案生成：输入卖点、参数、FAQ模板；输出结构化详情文案；审核事实准确性。
- 属性价格组装：输入 TikTok 模板 + 成本公式；输出可执行字段 JSON；审核枚举值/利润。
- 发布执行：输入最终 package；输出草稿链接/正式链接；审核发布结果与日志。

## 7. 必须补充给 Codex / OpenClaw 的附件清单
1. 主图规则文件：尺寸、背景、是否允许人物/手模、文案规范、禁用元素、参考案例。
2. 副图规则文件：九宫格中每一张图分别承担什么作用。
3. 标题规则文件：语言、字符上限、关键词优先级、品牌词是否可用、禁限词。
4. 详情页模板：固定模块顺序、每一模块字数上限、是否允许夸张表达。
5. A+ 图模板：每张图的标题字数、图中文字密度、风格参考。
6. TikTok 官方上传模板：类目属性、必填字段、枚举值、字段说明。
7. 价格公式文件：产品成本、包材、头程、尾程、平台费、税点、目标 ROI。
8. 飞书审批规范：谁审核、多久未处理提醒、驳回后如何重试。
9. 店铺映射表：站点、店铺名称、登录方式、默认物流模板、默认仓库、默认币种。

## 8. 风险点与防呆机制
- 图片不合规：主图先审；保留“重生成单图”能力。
- 标题违规：标题模板 + 长度校验 + 敏感词库。
- 属性错填：强制从枚举模板映射。
- 价格错误：公式计算，不允许 AI 直接定价。
- 详情与 A+ 不一致：先出详情结构，再出 A+ 图。
- 上架失败：失败截图 + 重试 + 人工接管。
- 批量污染：先单店单品灰度发布，再批量复制。

## 9. MVP 实施建议（最适合现在）
- 只做 1 个站点，例如 TH 或 ID。
- 只做 1 个类目，避免多类目属性复杂度。
- 只做 1 套主图模板 + 1 套详情模板。
- 只接 1 个上架入口，先能“写入草稿”就够。
- 一次只跑 1 个产品，跑通后再做批处理。
- 先把人工审核做扎实，再讨论无人值守。

## 10. 可直接给 Codex 的执行说明
- 以“任务状态机”方式实现：created -> media_pending -> media_approved -> copy_pending -> copy_approved -> package_ready -> publish_pending -> published / failed。
- 所有中间结果都落盘，并生成 `manifest.json`，记录文件路径、版本号、审核状态、错误日志。
- 飞书是状态切换器，不只是通知器。运营点击“通过/驳回”后，要真正更新任务状态。
- 图片、标题、详情、属性、价格必须分模块生成，禁止一个超长 prompt 一次性生成全部。
- 发布前必须运行 preflight：必填字段、图片数量、标题长度、币种、SKU 数量、价格下限、重量、类目枚举值。
- 发布动作失败时，输出可追溯证据：截图、错误信息、最后一步动作、重试次数。
