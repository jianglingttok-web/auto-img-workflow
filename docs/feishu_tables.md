# 飞书多维表设计

## 1. 目标

当前阶段先聚焦第一步：套图制作任务。

因此飞书建议拆成两张表：

- `生图任务表`：运营提交生图需求、图片审核、产物回写
- `项目进度表`：跨阶段汇总进度看板

原则：

- 生图入口只收集生图必需字段
- 后续上架字段不要提前污染生图表
- 多运营协作通过 `运营组 + 提交人 + 负责人 + 视图隔离` 实现

## 2. 生图任务表

### 推荐状态选项

`图片任务状态`：

- 待处理
- 生图中
- 待审核主图
- 待生成副图
- 待审核副图
- 重做中
- 已通过
- 已交付
- 异常
- 待人工处理

`图片审核状态`：

- 待审核
- 已通过
- 已打回

### 字段填写约束

- `产品卖点`：一行一个卖点，禁止写成长段说明。
- `参考链接`：一行一个链接，方便程序拆分。
- `合规要求`：一行一个约束，方便程序拆分。
- `营销表达`、`数字信息`：一行一条，便于 benefit_copy 模式读取。
- `主图数量`、`副图数量`：必须为非负整数。
- `主图数量 = 0` 且 `副图数量 = 0`：视为无效任务，直接拦截。
- `仅副图` 模式：必须提供 `已有主图/风格参考图`。
- `审核意见`：当 `图片审核状态=已打回` 时应必填。
- `主图预览`：回传时默认展示当前轮次主图第一张。
- `副图拼图预览`：回传时展示一张副图总览拼图，不在飞书里平铺全部副图。
- `完整图片包链接`：用于查看完整套图，不要求全部图片都直接塞进飞书消息。
- `异常原因`：当 `图片任务状态=异常` 时应必填。

### 任务模式自动判定

系统不要求运营手填 `任务模式`，而是由数量自动推断：

- `主图数量 > 0` 且 `副图数量 = 0`
  - `仅主图`
- `主图数量 = 0` 且 `副图数量 > 0`
  - `仅副图`
- `主图数量 > 0` 且 `副图数量 > 0`
  - `整套图`

### 首版最小字段集

如果要先跑最小闭环，只保留这些字段：

- 任务ID
- 提交时间
- 提交人
- 运营组
- 店铺
- 站点
- 产品名称
- 产品卖点
- 风格要求
- 产品白底图
- 使用图

- 副图数量
- 图片任务状态

## 4. 多运营协作建议

推荐结构：

- 一张统一底表
- 每个运营组一个专属视图
- 按 `运营组`、`提交人`、`负责人` 做筛选

不要给每个运营组复制一套完全独立的数据表，原因：

- 统计困难
- 字段容易漂移
- 后续 API 集成复杂

推荐至少保留三个区分字段：

- `运营组`
- `提交人`
- `负责人`

## 5. 与系统字段的映射思路

生图任务表只需要映射到第一阶段的 `product_brief.json`。

建议映射：

- `任务ID` -> `task_id`
- `产品名称` -> `product_name`
- `店铺` -> `shop_id`
- `站点` -> `target_market`
- `类目建议` -> `category_hint`
- `产品卖点` -> `selling_points`
- `风格要求` -> `style`
- `合规要求` -> `compliance_rules`
- `参考链接` -> `competitor_links`
- `营销表达` -> `marketing_phrases`
- `数字信息` -> `numeric_claims`
- `补充说明` -> `notes`
- `主图数量` + `副图数量` -> `image_rules`
- `产品白底图` / `使用图` / `已有主图/风格参考图` -> `reference_images`

当前阶段不建议把价格、SKU、属性模板强行塞进生图入口表。

## 6. 导入样例

仓库里已提供一份中文记录样例：

- `examples/feishu_image_task_record.sample.json`

可用以下命令导入：

```powershell
$env:PYTHONPATH = ".\src"
py -3 -m tk_listing_workflow.cli import-feishu-image-task --record .\examples\feishu_image_task_record.sample.json
```

导入后会在 `runtime/tasks/<任务ID>/product_brief.json` 写入标准化结果。
